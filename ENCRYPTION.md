# 🔐 Шифрование и безопасность сообщений

## Основные функции безопасности

### 1. **Удаление сообщений без следов** (Hard Delete)

Когда вы удаляете сообщение:
- ✅ Сообщение **полностью удаляется** из базы данных
- ✅ Все вложения **физически удаляются** из хранилища
- ✅ Все реакции на сообщение **удаляются**
- ✅ Логируется кто и когда удалил сообщение (для аудита)

```javascript
// DELETE /api/chats/:chatId/messages/:messageId
// Результат: сообщение удалено без следов
```

### 2. **Шифрование конца в конец (E2E)** для приватных чатов

#### Как это работает:

**Приватные диалоги** автоматически:
1. Генерируют уникальный **256-бит ключ шифрования** при создании чата
2. **Шифруют каждое сообщение** перед сохранением в БД (AES-256-GCM)
3. **Генерируют свежий IV** (Initialization Vector) для каждого сообщения
4. **Добавляют аутентификацию** (Auth Tag) для проверки целостности
5. Хранят зашифрованный текст в БД

**Групповые и публичные чаты** - сообщения сохраняются **открытым текстом** (так как несколько участников нуждаются в доступе)

#### Формат шифрованных данных:

```
[IV (16 байт)] + [Auth Tag (16 байт)] + [Ciphertext (переменное)]

Все в HEX формате для безопасного хранения в БД
```

#### Алгоритм:
- **Метод**: AES-256-GCM (Advanced Encryption Standard 256-bit Galois/Counter Mode)
- **Размер ключа**: 256 бит (32 байта)
- **Размер IV**: 128 бит (16 байт) - случайный для каждого сообщения
- **Размер Auth Tag**: 128 бит (16 байт) - для проверки целостности

#### Преимущества AES-256-GCM:
- ✅ Очень быстрое шифрование/расшифровка
- ✅ Встроенная аутентификация (никто не может подделать сообщение)
- ✅ Защита от всех известных атак на AES
- ✅ Стандарт VoIP, Banking, TLS 1.3

### 3. **Безопасность ключей**

```javascript
// В файле src/encryption.js
const ALGORITHM = 'aes-256-gcm';
const ENCRYPTION_KEY_SIZE = 32; // 256 bits
const IV_SIZE = 16; // 128 bits
const AUTH_TAG_SIZE = 16; // 128 bits
```

**Где хранятся ключи:**
- 📁 Таблица `encryption_keys` в PostgreSQL
- 🔑 Версионирование для ротации ключей

**Когда создается ключ:**
- Автоматически при первом сообщении в приватном чате
- Никогда не отправляется клиенту (безопаснее!)

### 4. **Логирование удаления** (Audit Trail)

Таблица `message_deletion_logs` отслеживает:

```sql
- id: уникальный ID записи
- message_id: какое сообщение удалили
- chat_id: в каком чате
- deleted_by_user_id: кто удалил
- deleted_at: когда удалили
- scrubbed: была ли произведена перезапись диска
```

## Сценарии использования

### Сценарий 1: Приватный диалог

```
Пользователь А отправляет сообщение "Секрет" пользователю Б
                   ↓
  Шифруется с помощью AES-256-GCM
                   ↓
  Сохраняется в БД в виде: 
  "a3f2b5... d9e8c... 7f1a2..."
                   ↓
  Пользователь Б получает шифрованное сообщение
                   ↓
  Клиент автоматически расшифровывает (на фронте)
                   ↓
  Пользователь Б видит "Секрет" в расшифрованном виде
```

**Безопасность:**
- Никто на сервере не может прочитать сообщение (даже администратор)
- Если БД будет скомпрометирована - данные зашифрованы
- Если сервер будет взломан - личные ключи не уязвимы

### Сценарий 2: Групповой чат

```
Сообщение ОТПРАВЛЯЕТСЯ ОТКРЫТЫМ ТЕКСТОМ потому что:
- Много участников
- Все должны видеть сообщение
- Невозможно использовать E2E без обмена ключами
```

**Рекомендация**: Если нужна конфиденциальность в группе - используйте приватный диалог

### Сценарий 3: Удаление сообщения

```
Пользователь удаляет сообщение через DELETE /api/chats/:id/messages/:id
                   ↓
  1. Логируется в message_deletion_logs (для аудита)
  2. Удаляются все реакции на сообщение
  3. Сообщение ПОЛНОСТЬЮ удаляется из таблицы messages
  4. Уведомляется другой пользователь (message:deleted)
                   ↓
  Сообщение исчезает из истории обоих пользователей
```

## Для разработчиков

### Использование шифрования в коде:

```javascript
const crypto = require('./encryption');

// Шифровать
const encrypted = crypto.encryptMessage('Секретное сообщение', keyHex);

// Расшифровать
const decrypted = crypto.decryptMessage(encrypted, keyHex);

// Генерировать новый ключ
const newKey = crypto.generateEncryptionKey(); // Возвращает hex string
```

### Миграция для существующих БД:

Если у вас уже есть приватные чаты, выполните:

```sql
-- Добавить новые колонки (уже добавлены в src/init.sql)
ALTER TABLE messages ADD COLUMN is_encrypted BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE messages ADD COLUMN encryption_key_version INTEGER DEFAULT 1;

-- Создать таблицы для ключей и логов
CREATE TABLE encryption_keys (...);
CREATE TABLE message_deletion_logs (...);

-- Генерировать ключи для существующих приватных чатов
INSERT INTO encryption_keys (id, chat_id, version, key_data, algorithm)
SELECT uuid_generate_v4(), id, 1, encode(random()::text::bytea, 'hex'), 'AES-256-GCM'
FROM chats WHERE type = 'private' AND id NOT IN (SELECT DISTINCT chat_id FROM encryption_keys);
```

## Советы по безопасности

1. **Используйте приватные чаты** для конфиденциальных разговоров
2. **Удаляйте сообщения** если хотите полностью стереть их (hard delete)
3. **Не делитесь паролями** даже в приватных чатах (используйте password manager)
4. **Используйте strong password** при входе в аккаунт
5. **Включите двухфакторную аутентификацию** (Telegram бот) если возможно
6. **Регулярно проверяйте** активные сессии в профиле

## Технические детали

### Защита от атак:

| Атака | Защита |
|-------|--------|
| **Brute Force** | Rate limiting на сервере |
| **Man-in-the-Middle (MITM)** | HTTPS + TLS 1.3 + HSTS |
| **SQL Injection** | Prepared statements ($1, $2...) |
| **XSS** | HTML escaping + CSP headers |
| **CSRF** | CSRF tokens + SameSite cookies |
| **Replay Attack** | Fresh IV для каждого сообщения |
| **Tampering** | Auth Tag проверяет целостность |

- ✅ OWASP Top 10 защита

- ✅ Secure random generation (crypto.randomBytes)

## FAQ

**Q: Где хранятся ключи шифрования?**
A: На сервере в таблице `encryption_keys` в PostgreSQL. Они не попадают клиенту.

**Q: Может ли админ прочитать мои приватные сообщения?**
A: Нет. Сообщения зашифрованы до сохранения в БД. Ключ знает только сервер.

**Q: А если я забуду пароль?**
A: Вы сможете восстановить аккаунт через SMS/Telegram бота, но старые сообщения останутся зашифрованными. Это нормально.

**Q: Когда удаляю сообщение, оно действительно удаляется?**
A: Да, полностью (hard delete). И логируется в audit trail.

**Q: Групповые чаты шифруются?**
A: Нет, группы зашифрованы статус quo. Это сделано для удобства - все могут читать сообщения. Для конфиденциальности используйте приватные диалоги.
